# 架构与产品决策记录

## 2026-05-27：旧材料只用于提取功能需求

**决定**：旧的 Markdown/TXT 文档仅用于确认系统想解决的问题和候选方向，不继承其中的 API、数据表、完成状态、技术细节或敏感配置。

**原因**：旧材料描述的是示例原型，且不同文档之间存在完成度和实现口径冲突。

**影响**：正式需求与开发契约统一维护在 `docs/`。

## 2026-05-27：示例代码允许完全重写

**决定**：旧应用代码、旧数据库与旧文档在需求提取完成后废弃并移出正式工程；正式实现按新契约重新建设。

**原因**：目标是构建符合正式需求、安全边界和可持续开发规范的新系统，而不是维护示例兼容性。

## 2026-05-27：首版采用 Python 模块化 MVC 单体

**决定**：正式版本使用 Python 后端，按 MVC 与业务模块组织为单体应用；首版不引入微服务。

**原因**：当前核心风险在于完成数据采集到智能问数的可靠闭环。单体能降低部署、分布式一致性和跨服务鉴权成本，并保持模块边界清晰。

**重新评估条件**：出现模块独立扩缩容、采集影响主站稳定性、独立团队交付边界或强隔离需求。

## 2026-05-27：首版范围锁定核心闭环

**决定**：首版包含认证/RBAC 与动态功能导航、模型引擎、智能瞭望、数据仓库、智能问数和必要安全审计。

**不包含**：即时通讯、数字员工、舆情大屏、语音/手势、复杂自动化和多数据库切换。

## 2026-05-27：新数据库与 API 文档为正式契约

**决定**：`docs/database/` 与 `docs/api/` 描述的是目标重写系统需要实现的接口和数据规则，后续代码应围绕这些契约开发。

**变更方式**：业务决定发生变化时，必须同步更新契约与本决策记录，而不是仅在代码中产生隐性差异。

## 2026-05-27：变更必须经过分支和 Code Review 流程

**决定**：正式变更采用 Conventional Commits；不得直接在 `main`、`master` 或其他受保护主分支提交；变更应在工作分支完成并由项目负责人 Code Review 后合并。

**原因**：项目将持续由人类与 AI 协作开发，需要通过可审查提交和人工合并门禁防止未经确认的设计或实现进入主线。

**影响**：AI 默认只能准备工作分支变更和评审材料，不得自行合并主分支；完整规则维护在 `docs/workflow/git.md`。

## 2026-05-27：以 MVP 文档和 Phase 计划组织三人协作

**决定**：新增 `docs/product/mvp.md` 作为首版验收边界，新增 `docs/planning/development-plan.md` 将实施拆分为可并行的 Phase 和三条责任流。

**原因**：正式需求描述能力范围，但三名开发者还需要明确交付判据、依赖顺序、并行边界与评审节点，才能减少共享文件冲突和重复设计。

**影响**：每项开发任务应关联一个 MVP 旅程和当前 Phase 工作包；Phase 完成以验收和 Code Review 合并为准。
