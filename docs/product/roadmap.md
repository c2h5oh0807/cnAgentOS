# 产品路线图

## 阶段 1：首版业务闭环

目标是提供可以实际使用和验收的核心链路。

| 工作包 | 交付内容 | 状态 |
| --- | --- | --- |
| 基础平台 | 认证、用户、角色、权限、动态功能导航、管理后台框架 | 待重写 |
| 模型能力 | 模型配置、默认模型、测试调用、调用记录 | 待重写 |
| 智能瞭望 | 数据源、规则、采集任务、执行结果 | 待重写 |
| 数据沉淀 | 内容存储、去重、筛选与可用状态治理 | 待重写 |
| 智能问数 | 会话、问题、检索依据、流式回答与引用 | 待重写 |
| 基础安全 | 敏感配置保护、采集边界、权限检查、审计 | 待重写 |

## 阶段 2：使用体验与运营能力

在首版真实使用反馈成立后评估：

- 数据统计和基础运营视图。
- 采集任务调度、失败重试和通知能力。
- 更丰富的来源解析适配能力。
- 问答质量评估、反馈与知识内容维护流程。

## 候选扩展池

以下来自早期对比分析，不是既定承诺：

| 方向 | 候选能力 | 启动前提 |
| --- | --- | --- |
| 即时通讯 | 私聊、群聊、文件收发、通讯录 | 产品明确需要实时协作场景 |
| 数字员工 | 校园/天气等 AI 助手、工具调用 | 问数主线稳定且有角色化需求 |
| 舆情与大屏 | 情感分析、风险识别、可视化 | 已积累足够有效数据与运营目标 |
| 交互增强 | 语音、手势 | 核心功能验收后有明确体验收益 |
| 自动化 | 定时采集、流程编排、告警 | 人工采集流程稳定且需要规模化 |
| 基础设施 | MySQL、多数据库、服务拆分 | 规模、部署或团队边界提出要求 |

## 变更规则

- 将候选能力升级为正式范围，必须记录到 `docs/context/decisions.md`。
- 新阶段开始前，需更新 `docs/context/current-status.md` 和相关数据库/API 契约。
