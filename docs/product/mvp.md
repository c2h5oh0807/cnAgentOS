# MVP 定义与验收

## 1. MVP 目标

MVP 要证明一条可实际演示、可继续扩展的核心链路成立：

```text
管理员配置权限与模型
-> 管理员配置安全的数据源并执行采集
-> 管理员治理入库内容
-> 用户基于可用内容发起问数
-> 系统流式返回回答和引用依据
```

MVP 完成不等于所有候选能力完成。它只验证智能瞭望与智能问数能够在基础安全和权限控制下闭环运行。

## 2. 必须交付的用户旅程

### 旅程 A：后台访问控制

1. 系统管理员登录后台。
2. 管理员创建或维护用户、角色与权限分配。
3. 管理员配置后台功能导航。
4. 不同角色用户只能看到并调用自己拥有权限的后台能力。

**通过标准**：页面导航与服务端权限同时生效；无权限请求被拒绝；关键变更有审计记录。

### 旅程 B：模型可用

1. 管理员新增一个 OpenAI 兼容模型配置并提交凭据。
2. 界面只展示凭据掩码。
3. 管理员完成模型连接测试和流式对话测试。
4. 管理员将已启用模型设为默认模型。

**通过标准**：测试正常产生脱敏调用记录；任意读取接口不返回模型密钥。

### 旅程 C：采集到数据治理

1. 管理员创建符合安全策略的数据源和采集规则。
2. 管理员发起一次手动采集任务。
3. 系统展示任务成功、失败或部分失败状态。
4. 管理员查看采集内容，并调整可用于问数的治理状态。

**通过标准**：有效内容入库且可追溯；重复内容按规则处理；非法目标被 SSRF 控制拒绝。

### 旅程 D：有依据的智能问数

1. 已授权用户新建问数会话。
2. 用户提交与可用内容相关的问题。
3. 系统以 SSE 返回回答。
4. 页面展示回答引用的内容标题、来源与摘录。
5. 用户重新打开会话仍可查看历史回答与引用。

**通过标准**：回答只引用可用内容；引用可追溯；模型失败时保留可诊断的脱敏失败状态。

## 3. MVP 功能清单

| 能力 | MVP 必须包含 | 对应契约 |
| --- | --- | --- |
| 认证与会话 | 登录、退出、当前用户、会话撤销与 CSRF | `docs/api/auth-and-rbac.md` |
| RBAC 与导航 | 用户/角色管理、权限分配、动态导航树、服务端授权 | `docs/api/auth-and-rbac.md` |
| 模型引擎 | 脱敏配置、连接测试、默认模型、SSE 测试、调用统计 | `docs/api/model-engine.md` |
| 智能瞭望 | 数据源、规则、手动任务、任务状态与 SSRF 防护 | `docs/api/watch-and-data.md` |
| 数据仓库 | 内容入库、去重追踪、筛选、状态治理 | `docs/api/watch-and-data.md` |
| 智能问数 | 会话、依据检索、SSE 回答、引用展示与历史记录 | `docs/api/question-answering.md` |
| 横向能力 | 密钥加密/脱敏、高风险审计、错误边界 | `docs/architecture/security.md` |

## 4. 不纳入 MVP

- 普通用户自助注册，除非后续明确升级为首版需求。
- 自动定时采集、失败重试调度或流程编排。
- 即时通讯、数字员工、舆情分析与大屏。
- 语音、手势、多数据库或微服务拆分。
- 复杂搜索优化、向量化检索或推荐算法；MVP 只要求可用的数据检索依据能力。

## 5. MVP 质量门槛

| 维度 | 通过条件 |
| --- | --- |
| 功能 | 四条用户旅程均可在测试环境完整演示 |
| 权限 | 后台受保护接口具备服务端权限测试，无权用户无法绕过导航访问 |
| 安全 | 密钥不明文暴露；CSRF 与 SSRF 关键路径具备验证；敏感日志无泄露 |
| 数据 | 采集内容、治理状态、问答引用和审计信息可追踪 |
| 接口 | 实现符合 `docs/api/` 和 `docs/database/` 的正式契约，偏离已记录决策 |
| 交付 | 变更以工作分支和 Conventional Commits 提交，经项目负责人 Code Review 后合并 |

## 6. MVP 演示脚本

1. 以系统管理员登录，配置角色权限与后台导航可见性。
2. 创建模型配置、执行流式测试，并设为默认模型。
3. 新建安全数据源与解析规则，执行一次采集任务。
4. 在数据仓库确认结果，标记至少一条内容为可用于问数。
5. 以具备 `qa.use` 权限的用户提问，展示流式回答及引用。
6. 展示调用日志和高风险操作审计记录，验证秘密字段已脱敏。
