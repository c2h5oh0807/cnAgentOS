# Git 与 Code Review 工作流

## 1. 基本原则

- 采用 Git 管理正式代码与文档变更；若工程尚未初始化 Git，在开始正式实现前初始化仓库并建立受保护的主分支。
- 主分支指 `main`，若仓库历史使用 `master` 或其他受保护默认分支，则同样适用本规则。
- 不允许 AI 或开发者直接在主分支上提交功能、修复或文档变更。
- 所有变更通过工作分支提交，并由项目负责人完成 Code Review 后合并。
- AI 不得自行合并 PR、直接推送主分支、绕过评审或覆盖未评审历史。
- 三人并行任务需遵守 `docs/planning/development-plan.md` 中的工作包边界；修改共享契约或共同文件前先同步协作者。

## 2. 分支流程

开始修改前：

1. 检查当前工作树和当前分支，避免覆盖已有未提交改动。
2. 若位于主分支，基于最新主分支创建工作分支后再提交变更。
3. 若任务已经位于专用工作分支，在确认变更范围一致后继续工作。

分支命名建议：

| 类型 | 命名示例 |
| --- | --- |
| 新功能 | `feat/watch-sources` |
| 修复 | `fix/model-credential-mask` |
| 文档 | `docs/ai-context` |
| 重构 | `refactor/mvc-bootstrap` |
| 构建或工具 | `chore/project-setup` |

分支可按计划工作包进一步命名，例如 `feat/phase-1-auth-rbac`、`feat/phase-2-collection`、`feat/phase-3-qa-ui`。

## 3. 约定式提交

提交信息遵循 Conventional Commits：

```text
<type>(<scope>): <description>
```

| type | 适用场景 |
| --- | --- |
| `feat` | 新增用户可见或业务能力 |
| `fix` | 修复缺陷或安全问题 |
| `docs` | 只修改文档 |
| `refactor` | 不改变功能行为的结构调整 |
| `test` | 添加或修正测试 |
| `chore` | 工具、依赖或工程维护 |
| `build` | 构建与打包配置 |
| `ci` | 持续集成配置 |

要求：

- 描述使用简短明确的祈使表达，说明本次提交的单一目的。
- 一个提交只包含可独立审查的一组相关变更；不要混入无关格式化、临时文件或秘密配置。
- 有破坏性接口或数据契约变更时，在提交正文或 footer 中标记 `BREAKING CHANGE:` 并同步更新正式文档。

示例：

```text
docs(workflow): define branch and review policy
feat(watch): add source management endpoints
fix(security): reject private network collection targets
```

## 4. 提交前检查

在工作分支创建提交前，必须：

1. 检查变更范围，排除无关文件、构建产物、秘密和本地配置。
2. 更新本任务影响的需求、API、数据库或动态上下文文档。
3. 运行与变更相关的测试、静态检查或人工验收。
4. 确认提交信息符合约定式提交格式。

## 5. Code Review 与合并

- 完成任务后，应将工作分支变更提供给项目负责人审查，可通过 PR 或本地分支差异进行 Review。
- 评审材料应包含：变更摘要、受影响文档/模块、验证结果、已知风险及契约变更说明。
- Code Review 提出的修改继续提交到同一工作分支，并重新提供验证结果。
- 只有项目负责人明确审核通过后，变更才可以合并至主分支。
- 合并操作由项目负责人执行，或在其明确授权的情况下执行；AI 默认不得执行合并。

## 6. AI 行为约束

- AI 在需要提交代码前必须先确认当前不在主分支；若处于主分支，应创建工作分支或请求用户指定分支。
- AI 只有在用户明确要求提交、推送或创建 PR 时才执行对应操作。
- AI 不得运行破坏历史或绕过审查的操作，如对主分支强推、擅自合并、删除用户未确认的工作分支。
- AI 最终回复应说明变更是否已提交、所在分支、执行的验证，以及是否等待 Code Review。
