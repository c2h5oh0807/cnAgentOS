# AI 开发工作流

## 1. 文档优先级

开始开发时，按以下顺序理解任务：

1. 用户本轮明确要求。
2. `docs/product/requirements.md` 中的正式需求。
3. `docs/product/mvp.md` 中的交付边界与验收标准。
4. `docs/planning/development-plan.md` 中的当前 Phase、依赖和并行工作包。
5. `docs/context/decisions.md` 中的已锁定选择。
6. `docs/architecture/`、`docs/database/` 和 `docs/api/` 中的开发契约。
7. `docs/context/current-status.md` 中的进度。
8. `docs/workflow/git.md` 中的分支、提交与评审规则。
9. 如用户另行提供的参考资料，仅可用于需求澄清，不能覆盖正式契约。

发生冲突时，不得以原型实现覆盖正式契约；应修订契约或向用户确认产品变更。

## 2. Python 环境与依赖管理

- 项目统一使用 `uv` 管理 Python 版本、依赖、虚拟环境和锁文件；`pyproject.toml` 与 `uv.lock` 均应纳入评审与版本控制，`.venv/` 不提交。
- 首次开始工作或依赖变化后执行 `uv sync`，确保本地环境与锁文件一致。
- 新增和移除依赖使用 `uv add <package>` 与 `uv remove <package>`；依赖变更必须同时提交对应的 `pyproject.toml` 和 `uv.lock` 变化。
- Python 程序、测试、格式化、迁移与启动命令统一通过 `uv run ...` 执行，避免绕过项目环境。
- 提交或 CI 验证中使用 `uv lock --check` 检查依赖声明与锁文件一致性；除明确的部署集成需求外，不引入另一套依赖真源。

## 3. MVC 实现规则

- 按业务模块组织代码，并维持 Controller、Model、View 三层边界。
- Controller 处理输入校验、身份与权限入口、调用 Model、组装 HTTP/SSE 响应。
- Model 承担业务规则、事务、外部模型/采集调用和数据访问；复杂时可在 Model 内分为领域服务与仓储实现。
- View 负责页面交互和数据显示；不得依靠隐藏控件保护后端操作。
- 横向安全能力和审计规则须在各受影响模块一致执行。

## 4. 单次任务流程

### 分析

1. 阅读入口和本任务关联的需求、契约、状态、问题文档。
2. 确认本任务所属 MVP 旅程、开发 Phase 和工作包负责人，避免多人修改同一责任边界。
3. 若仓库已启用 Git，检查当前分支和工作树；不得直接在主分支提交实现变更。
4. 检查正式工程中的相关代码；新实现按正式契约建设，不假定早期原型已正确实现。
5. 确认本任务是否改变需求、数据模型或 API 契约。

### 设计与实现

1. 若改变契约，先同步更新相关文档和决策。
2. 实现最小完整业务行为，避免顺手加入候选路线图能力。
3. 为权限、敏感配置、外部调用和状态迁移保留验证与失败处理。
4. 建立与变更风险匹配的测试或验收验证。

### 收尾

1. 运行可用的验证命令并记录结论。
2. 更新 `docs/context/current-status.md` 的正式完成度。
3. 新的长期决策写入 `docs/context/decisions.md`。
4. 风险、阻塞或暂缓项目写入 `docs/context/issues.md`。
5. 可供未来任务重复使用的经验写入 `docs/context/ai-memory.md`。
6. 用户要求提交或交付时，在工作分支使用约定式提交，并提供给项目负责人 Code Review；未获明确通过不得合并主分支。

## 5. 数据库与 API 变更

- 正式 schema 变更必须维护迁移方案，不允许仅修改运行时自动建表逻辑。
- API 变更必须更新模块接口文档，并兼顾权限、错误格式和敏感字段响应约束。
- 首版实现若需要偏离已定义契约，应将偏离视为决策变更并记录原因。

## 6. 安全检查清单

- 是否引入或暴露了凭据、密码、Cookie、令牌等秘密。
- 是否为受保护操作增加了服务端权限校验与 CSRF 防护。
- 是否对外部 URL、重定向、内容大小和失败信息实施采集安全控制。
- 是否对模型输入中的外部内容按不可信数据处理。
- 是否为高风险管理动作记录了脱敏审计信息。
